# ADR-001: Surface AI governance as an operator dashboard

## Status

Accepted

## Date

2026-06-04

## Context

The service desk already records tickets, agent memory, human approvals, audit events, and trace spans. Operators can inspect parts of this information inside an individual ticket, but there is no consolidated surface that answers: where is AI risky, where is human review blocking work, and what improvement should the team prioritize next.

The source PDF at `C:\Users\veron\Downloads\arquitetura_de_solucoes_com_ia.pdf` has 220 scanned pages and no extractable text through the available local tooling. OCR tools are not installed, so this cycle uses the repository documentation, current implementation, and market research as source context.

## Decision

Create a frontend AI governance and improvement dashboard that reuses existing ticket, audit, and trace data. The dashboard will summarize approval pressure, low-confidence AI decisions, trace health, and recommended operational improvements. It will not introduce a new backend subsystem in this cycle.

## Alternatives Considered

### Alternative 1: Backend recommendation engine first

- **Description**: Add a backend service that computes and persists improvement recommendations.
- **Pros**: Easier to expose recommendations across clients later.
- **Cons**: Requires new domain contracts before the UI value is validated.
- **Why rejected**: Current ticket and trace data is already available to the frontend, so the first version can stay lean.

### Alternative 2: Only improve ticket detail panels

- **Description**: Add more governance data to the existing ticket workspace.
- **Pros**: Very low navigation impact.
- **Cons**: Does not help managers find portfolio-level improvement opportunities.
- **Why rejected**: The user asked for a loop of discovering improvements, so the first feature must expose cross-ticket improvement signals.

## Consequences

### Positive

- Operators get a single place to inspect AI health and improvement priorities.
- The implementation reuses existing endpoints and avoids new persistence risk.
- Future cycles can promote the recommendation model into backend logic if needed.

### Negative

- Recommendations are heuristic in the first version.
- The dashboard depends on current in-memory trace and audit retention.

### Risks

- Recommendations may look too authoritative. The UI will label them as operational recommendations and show evidence counts.
- Large queues may require server-side aggregation later.

## Implementation Notes

Add a frontend view, API client function for `/api/agents/runs`, navigation wiring, CSS, and an e2e assertion. Keep the implementation deterministic and testable without external AI keys.
