# ADR-001: Expose deterministic agent evals in AI governance

## Status

Accepted

## Date

2026-06-04

## Context

The backend already contains a deterministic service desk agent eval suite covering intake readiness, ticket outcome, RAG grounding, approval gates, and workflow trajectory. The governance dashboard shows live tickets, traces, audit events, feedback, and RAG coverage, but it does not expose whether curated agent behavior still passes executable checks.

External AI operations guidance points in the same direction: production AI governance should combine tracing, evaluation, monitoring, and human feedback in a continuous control loop. MLflow describes AI monitoring around traces, automatic quality evaluation, human feedback, and scorers. Grafana documents GenAI evaluation dashboards for model quality and safety. DataRobot frames agentic AI readiness as evaluation, monitoring, and governance across the lifecycle.

## Decision

Expose a deterministic eval report through a backend endpoint and render a compact eval quality panel inside the existing AI governance view.

The endpoint will run the curated service desk eval suite in an isolated in-memory harness using deterministic fallback behavior. The frontend will request the report with the rest of the workspace data and show pass rate, aggregate score, case status, and failed scorers if any.

## Alternatives Considered

### Frontend-only heuristic

Use current tickets, traces, and feedback to infer agent quality in the browser.

- Pros: no backend execution path.
- Cons: does not reuse existing eval cases or scorers, and risks presenting operational symptoms as test coverage.
- Rejected because the project already has executable eval logic.

### Static CI artifact only

Generate a report during CI and display the latest committed artifact.

- Pros: stable and cheap at runtime.
- Cons: requires artifact hosting and does not work in the local app without another integration.
- Rejected for this MVP because the app can run a small deterministic suite on demand.

### Persisted eval history

Store every eval run and trend it over time.

- Pros: supports drift analysis.
- Cons: adds persistence, retention, and UI complexity.
- Rejected for this cycle. A live snapshot is enough to close the current visibility gap.

## Consequences

- Positive: managers can see executable AI quality evidence in the same place as traces, feedback, and RAG health.
- Positive: backend tests and UI smoke checks can verify the eval report path.
- Trade-off: the report is a snapshot, not a historical trend.
- Risk: running evals on every refresh could become expensive if cases grow. Mitigation: keep the suite deterministic and small in this cycle.
