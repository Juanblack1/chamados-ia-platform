# ADR-001: Add client-side operational queue filters

## Status

Accepted

## Date

2026-06-04

## Context

The production service desk plan says analysts should scan the queue by SLA risk, priority, status, assigned group, requester, service, and AI confidence. The current queue supports text search and status filtering, while the table already contains priority, SLA, and AI confidence data.

Service desk products commonly expose queue views and filters around priority, SLA, and assignment because these are the fields analysts use to decide what to work next. Zendesk documents ordering by time to next SLA breach for queue prioritization. Gorgias documents priority as a view/filter condition. LiveAgent emphasizes using priority and SLA management to drive ticket handling.

## Decision

Add client-side filters for priority, SLA risk, and AI confidence in the existing topbar when the queue is active.

The filters will apply to the already loaded ticket list together with existing status and text search filters. No backend endpoint or persistence is needed in this cycle.

## Alternatives Considered

### Server-side query filters

- Pros: scales better for very large queues.
- Cons: requires API changes before the product has large data volume.
- Rejected for this cycle.

### Full saved queue views

- Pros: powerful for teams and roles.
- Cons: needs persistence, naming, ownership, and permissions.
- Rejected because the first need is fast operational narrowing.

### Table column sorting only

- Pros: simple UI.
- Cons: sorting does not isolate breached SLA, critical priority, or low-confidence work.
- Rejected because analysts need filters, not just order.

## Consequences

- Positive: analysts can isolate critical work faster.
- Positive: no backend risk and no schema change.
- Trade-off: filters apply only to the loaded authorized queue.
- Risk: topbar can become crowded. Mitigation: compact select controls and responsive wrapping.
